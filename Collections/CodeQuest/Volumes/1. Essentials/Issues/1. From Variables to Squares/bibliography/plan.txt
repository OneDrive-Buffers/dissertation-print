# Polish Pass for Issue 1 Toward an 80-Page Ready-to-Ship Build

## Summary
Rebuild the moved Issue 1 into a cohesive, print-ready edition with a stronger programming/editor visual identity, much cleaner Romanian copy, visibly present Blockly experiments, QR-linked simulator placeholders, and a rebalanced pedagogy that gives more room to control structures. The product target for this iteration is 78-82 pages, with a preferred landing at 80, and no standalone appendices.

   9. Structura `pentru`: 8 pages

   12. Pong explained in detail: 6 pages
   13. TicTacToe final challenge: 6 pages
   14. Final submission page + back cover: 2 pages

- Keep one continuous game-logic narrative under the surface of the whole magazine: state in memory, score and lives, movement and counters, decisions at doors/collisions, timers and waves, then board/grid logic, Pong, and finally TicTacToe.
- Rewrite all body copy for fluent Romanian with diacritics, while keeping pseudocode and language syntax ASCII where that reads more naturally.
- Remove robotic framing, “this issue presents” filler, and any pejorative or careless pedagogical wording; the voice should stay warm, precise, and game-oriented.
- Normalize terminology everywhere: `daca`, `altfel`, `cat timp`, `pentru` in code and pseudocode, with no `catTimp` and no glued or oddly capitalized structure names.
- Normalize all `.tex` source content to UTF-8-safe Romanian and explicitly clean up mojibake or broken character sequences before content polishing.
- Rebuild the visual system so backgrounds vary by page type instead of using one repeated global pattern. Use five page themes: `story`, `theory`, `guided`, `simulation`, and `challenge`.
- Replace literal UI filler such as `lazy-nvim://logic/session` with abstract editor-inspired chrome only. The aesthetic can feel terminal/Vim/VSCode-like without printing fake protocol strings.
- Add real Blockly-style rendered blocks inside the lessons, not just faint ghost ornaments. Every major concept chapter must include at least one visible Blockly experiment panel.
- Add simulator cards with QR placeholders and stable slugs for the concepts that most need visualization. Use these slugs in this pass: `cq-i1-ram-cells`, `cq-i1-variable-state`, `cq-i1-operator-score`, `cq-i1-if-branch`, `cq-i1-while-counter`, `cq-i1-for-interval`, `cq-i1-grid-scan`, `cq-i1-tictactoe-board`.
- Every simulator page must contain four fixed elements: what the simulator shows, what control the pupil should move, one prediction question before scanning, and one “incearca si observa” prompt after scanning.
- Remove standalone appendix sections completely. Fold their useful content into inline quick-check comparison tables with the same shape each time: `Pseudocod | C++ | Python | Java`.
- Put those inline syntax tables directly inside `variabile`, `operatori`, `daca`, `cat timp`, `pentru`, and `structuri imbricate`, never as detached appendix blocks.
- Rebuild the control-structures opening as a true 2-page spread with no normal header treatment, one continuous visual flow across the gutter, and a graphic that visibly runs off the second page.
- Expand `daca` content to cover decision points, condition quality, one-branch and two-branch logic, short traces, and game examples such as key/door, collision, checkpoint, and game over.

## Public Interface / Macro Surface
- Add a page-theme switch macro with this exact shape: `\SetPageTheme{story|theory|guided|simulation|challenge}`.
- Add a simulator callout macro with this exact shape: `\SimulatorCard{slug}{titlu}{ce-vezi}{ce-incerci}`.
- Add a Blockly experiment macro with this exact shape: `\BlocklyExperiment{slug}{titlu}{descriere}`.
- Add an inline language comparison macro with this exact shape: `\SyntaxCheck{concept}`.
- Use these macros consistently so later issues can reuse the same print system without redesigning the layout layer.
